# Приложение для создания списка желаний в соцсетях — план развития

## Задача
Разработать веб-приложение для создания списка желаний в соцсетях, работающее в производственной среде, для выполнения тестового задания.

Пользователь создает список желаний, добавляет подарочные товары и делится общедоступной ссылкой.

Друзья могут зарезервировать подарки или внести деньги на покупку дорогих товаров.

Основное правило конфиденциальности:
владелец списка желаний не должен видеть, кто зарезервировал подарок или кто внес деньги.

Основное техническое правило:
приложение должно поддерживать обновления в реальном времени без перезагрузки страницы.

---

## Правила определения объема работ

Мы придерживаемся подхода MVP (Most Proof of Person) в первую очередь.

Разрешено:
- только функции, необходимые для выполнения задания
- минимальные улучшения UX, снижающие трение
- безопасные инженерные решения, повышающие стабильность

Не разрешено:
- дополнительные модули, не относящиеся к заданию
- преждевременная оптимизация
- ненужное расширение архитектуры
- дополнительные функции до завершения основных потоков

## Требования к продукту

### R1. Учетная запись пользователя
Пользователь может зарегистрироваться и войти в систему, используя электронную почту и пароль.

### R2. Управление списком желаний
Владелец может:
- создать список желаний
- установить заголовок и описание
- добавить товары
- редактировать товары
- удалить товары

### R3. Данные о товаре
Каждый товар поддерживает:
- название
- URL
- цену
- изображение

### R4. Публичный доступ
Каждый список желаний имеет публичную ссылку.
Посетители могут открыть его без регистрации.

### R5. Резервирование подарка
Посетитель может зарезервировать подарок, чтобы он не дублировался.

### R6. Групповые пожертвования
Дорогие подарки могут принимать пожертвования от нескольких человек.
Индикатор прогресса показывает, сколько собрано.

### R7. Обновления в реальном времени
Все пользователи мгновенно видят обновления резервирования и пожертвований.

### R8. Защита от неожиданностей
Владелец может видеть статус и итоги, но не может видеть личности зарезервировавших или пожертвовавших.

---

## Решения по продукту

### D1. Два режима работы товара
Каждый товар поддерживает один из двух режимов:

1. single_gift
- один человек резервирует товар

2. group_gift
- много людей вносят вклад в целевую сумму

Причина:
это четко соответствует заданию и упрощает пользовательский опыт.

### D2. Публичная страница без принудительной аутентификации
Посетители могут просмотреть список желаний по публичной ссылке без регистрации.

Причина:
меньше препятствий и соответствует требованиям задания.

### D3. Владелец видит только сводные данные
Владелец видит:
- зарезервировано / не зарезервировано
- собранную сумму
- количество участников

Владелец не видит:
- имена резервирующих
- имена участников
- суммы индивидуальных взносов

Причина:
сохранение эффекта неожиданности.

### D4. Макет, ориентированный на мобильные устройства
Публичная страница и панель управления должны корректно отображаться на мобильных устройствах.

Причина:
явное требование и общая проверка рецензентов.

---

## Крайние случаи, которые необходимо поддерживать

Приложение должно безопасно обрабатывать:

- пустой список желаний
- отсутствующее изображение товара
- отсутствующую цену товара
- неработающую или удаленную внешнюю ссылку
- уже зарезервированный один подарок
- одновременные попытки резервирования
- сумму взноса, превышающую оставшуюся целевую сумму
- владельца, открывающего публичную ссылку на свой список желаний
